Smart Secure Motion Lock

A CC3200-based embedded security system with motion detection, password-controlled arming/disarming, and AWS IoT cloud alerts.

Zijian Li • UC Davis

Project Overview

This project implements a smart embedded security system using the TI CC3200 LaunchPad. The system monitors motion through an accelerometer, allows the user to arm or disarm the system using an IR remote password, and sends cloud alerts through AWS IoT when suspicious motion is detected.

The goal of the project was to design a reproducible prototype that combines sensing, user interaction, state-machine-based control, and wireless cloud integration into one embedded system. The design uses multiple communication mechanisms, including I2C for sensor communication, IR signal decoding for password entry and commands, and Wi-Fi for remote alert publishing.

The final prototype demonstrates both local embedded behavior and connected-device functionality. It supports states such as disarmed, armed, monitoring, and alert. When motion exceeds a threshold while the system is armed, the board triggers a cloud notification workflow and provides local feedback to the user.

Functional Specification

Figure 1. High-level state machine of the Smart Secure Motion Lock.

The system starts in a disarmed or idle state. The user enters a password using the IR remote to arm the system. Once armed, the board transitions into a monitoring state and continuously checks the accelerometer readings.

If the measured motion exceeds the selected threshold, the device enters an alert state. In this state, it generates local feedback and publishes an alert message to AWS IoT. The system may then be reset or disarmed through the correct remote input.

This state-machine design makes the project easier to debug, explain, and reproduce.

System Architecture

Figure 2. System architecture and major communication paths.

The CC3200 LaunchPad serves as the central controller of the system. The accelerometer communicates with the CC3200 through the I2C interface and provides motion data. The IR receiver captures input from the remote and allows password entry or user commands.

When an alert is triggered, the CC3200 uses its Wi-Fi interface to publish a message to AWS IoT. A cloud rule can then forward the event to SNS or another notification service, allowing the user to receive an email alert. Local feedback is provided through LEDs and optional display output.

Implementation

Hardware Implementation

The hardware setup is centered around the TI CC3200 LaunchPad. The accelerometer is connected through I2C and is sampled to measure motion. The IR receiver is connected to a GPIO pin and decoded in software based on pulse timing. Optional visual indicators such as LEDs or an OLED display are used to show the system state.

Software Implementation

The software is structured around a state machine that controls initialization, arming, monitoring, alert handling, and reset behavior. The program configures the sensor interface, reads motion values, decodes remote input, and determines whether an alert condition should be triggered.

Cloud Integration

AWS IoT is used as the cloud endpoint for alert reporting. The CC3200 connects using configured device credentials and publishes alert messages when motion is detected. AWS rules can then route these messages to SNS email notifications.

Engineering Focus

A major goal of the project was to create a clean and reproducible embedded system that combines sensing, communication, remote notification, and clear system behavior in one unified design.

Reproducibility

This project was documented with reproducibility as a primary goal. A student attempting to rebuild the system would need the following:

  • TI CC3200 LaunchPad
  • Accelerometer module
  • IR receiver and compatible IR remote
  • Breadboard and jumper wires
  • AWS IoT Thing, certificates, and policy configuration
  • Code Composer Studio / TI SDK environment

To reproduce the system, the hardware components are first connected to the CC3200. The embedded software is then configured for sensor reading, IR decoding, and state transitions. After that, AWS credentials and endpoint information are added so the board can publish alert messages. With both hardware and cloud setup completed, the system can be tested by arming the device and generating motion to trigger an alert.

Challenges

One major challenge was integrating multiple subsystems into a single embedded project. The accelerometer required stable I2C communication, the IR receiver depended on accurate timing for signal decoding, and AWS connectivity required careful certificate and network setup.

Another challenge was selecting motion thresholds that were sensitive enough to detect real movement without triggering too many false alerts. Additional debugging effort was also needed to ensure that sensing, user input, and cloud publishing all worked reliably together.

Future Work

Future improvements could include a more polished local display interface, event history storage, support for multiple passwords or users, and more advanced motion classification to distinguish between small vibrations and meaningful security events. The system could also be extended with additional sensors such as door switches, microphones, or cameras.
